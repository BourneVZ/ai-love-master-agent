# 项目开发宪法
# Version: 1.0, Ratified: 2026-06-19

本文件定义了本项目不可动摇的核心开发原则。所有 AI Agent、开发者在进行需求分析、技术规划、代码实现、测试设计和缺陷修复时，必须无条件遵循。

本宪法适用于当前项目技术栈：

- 后端：Java 21、Spring Boot 3、Spring AI
- 前端：Vue 3、Vite、Axios、SSE
- AI 能力：Tool Calling、RAG、MCP、ReAct Agent

---

## 第一条：简单性原则 (Simplicity First)
**核心：** 优先采用与当前技术栈一致、最直接、最少抽象的实现方式。绝不为“未来可能需要”而提前设计复杂结构。

- **1.1 (YAGNI):** 只实现 `spec.md` 中明确要求的功能，或当前代码变更直接依赖的最小配套能力。
- **1.2 (框架内优先):** 后端优先使用 Spring Boot / Spring AI 已提供的能力；前端优先使用 Vue 3 生态内现有方案。除非现有框架无法满足需求，否则不得额外引入新框架。
- **1.3 (已有模式优先):** 新功能必须优先复用项目现有模式，如 `Controller -> App/Agent -> Tool` 的分层、SSE 输出方式、集中工具注册方式。
- **1.4 (反过度工程):** 简单类、简单方法、简单数据结构优于过度设计的接口层、工厂层和抽象继承层。只有在项目中已出现重复逻辑且复用收益明确时，才允许抽象。
- **1.5 (依赖克制):** 新增依赖必须有明确必要性。能用 JDK、Spring、Vue 现有能力解决的问题，不得为了“更优雅”引入第三方库。

---

## 第二条：测试优先原则 (Test-First Imperative)
**核心：** 所有新功能、行为变更和缺陷修复，都必须先定义可失败的验收方式，再实现代码。

- **2.1 (先验收后实现):** 编码前必须先明确验收标准。优先补充或更新测试用例，再进行实现；至少要先形成可验证的测试 Case 或断言目标。
- **2.2 (Red-Green-Refactor):** 对可自动化验证的改动，应遵循“先失败、再通过、后重构”的基本循环。
- **2.3 (贴近真实依赖):** 优先编写集成测试，尽可能复用 Spring Boot Test、真实 Bean 装配、真实 HTTP/SSE 链路和真实工具调用流程。
- **2.4 (Mock 克制):** 仅在外部依赖不可控、成本过高或不稳定时使用 Mock。不得为了凑单元测试覆盖率而把真实链路切碎到失去业务意义。
- **2.5 (测试分层):** 
  - 工具类优先覆盖成功路径与异常路径。
  - 应用层优先覆盖多轮对话、RAG、工具调用、MCP 调用。
  - 前端优先覆盖路由跳转、SSE 消息消费、连接异常处理。
- **2.6 (缺陷必有回归用例):** 每个已修复 Bug 都必须补一个能稳定复现该问题的回归测试，或在无法自动化时补充明确的手工测试 Case。

---

## 第三条：明确性原则 (Clarity and Explicitness)
**核心：** 代码的首要目标是让维护者快速理解系统行为、依赖关系和失败方式。

- **3.1 (显式错误处理):** 所有异常都必须被显式处理。禁止吞异常、空 `catch`、仅打印不返回失败信息。
- **3.2 (错误语义清晰):** 
  - 后端 Java 代码应优先保留原始异常原因，必要时包装为带上下文信息的异常。
  - 工具层若按当前项目约定返回字符串错误，错误文本必须明确说明失败动作与原因。
  - SSE 链路发生异常时，必须明确结束连接或向前端暴露错误状态。
- **3.3 (显式依赖注入):** 禁止使用隐式全局状态传递核心业务依赖。模型、工具、向量库、配置项等必须通过 Spring 注入或构造参数显式传入。
- **3.4 (边界清晰):** Controller 只负责协议和接口暴露；业务编排放在 `LoveApp`、`YuManus` 等应用层/Agent 层；工具逻辑放在独立 Tool 类中。
- **3.5 (命名直白):** 类名、方法名、字段名必须直接反映业务含义。禁止使用无法体现职责的模糊命名。
- **3.6 (注释克制但有效):** 只在逻辑不自明、AI 编排链路或工程约束容易误解时添加注释。注释必须解释“为什么”，而不是重复“做了什么”。

---

## 第四条：架构解耦原则 (Decoupling by Default)
**核心：** 任何新增功能都不得破坏当前项目已形成的前后端分离、能力分层和增强模块可插拔结构。

- **4.1 (前后端解耦):** 前端只能依赖稳定 HTTP / SSE 接口，不得耦合后端内部类结构、Prompt 细节或工具注册实现。
- **4.2 (AI 能力可插拔):** 基础聊天、RAG、Tool Calling、MCP 必须保持可独立启停，不得把某一增强能力强绑到所有对话链路。
- **4.3 (工具集中注册):** 所有工具能力必须统一注册、统一注入，禁止在业务逻辑中随意创建分散的工具实例。
- **4.4 (供应商隔离):** 大模型提供商、搜索服务、向量库和 MCP 服务都属于外部依赖，相关配置必须集中在配置文件或可注入组件中，不得散落硬编码。
- **4.5 (协议优先):** 流式输出、消息结构、工具返回结果的消费逻辑必须优先围绕接口契约设计，而不是围绕单次实现细节设计。

---

## 第五条：安全与约束原则 (Safety and Guardrails)
**核心：** 本项目包含终端执行、文件写入、资源下载等高风险能力，任何实现都必须先考虑边界与风险。

- **5.1 (高风险工具谨慎开放):** `TerminalOperationTool`、文件操作、下载工具默认视为高风险能力。新增或修改相关逻辑时，必须优先考虑目录限制、参数校验、超时控制和最小权限。
- **5.2 (配置不入库):** 真实 API Key、数据库密钥、外部服务凭据不得硬编码进源码、测试代码或文档示例。
- **5.3 (外部输入不可信):** 用户输入、网页内容、下载地址、命令参数、MCP 返回值都必须被视为不可信输入。
- **5.4 (失败优于误执行):** 当外部依赖不可用、参数不合法或风险无法判断时，系统应优先拒绝执行或返回明确失败，而不是“猜测性执行”。

---

## 第六条：演进一致性原则 (Consistency in Evolution)
**核心：** 新增代码必须与当前仓库的组织方式、测试习惯和运行方式保持一致，不得引入风格割裂的“平行体系”。

- **6.1 (遵循现有目录结构):** 后端代码放在现有 `controller`、`app`、`agent`、`tools`、`rag`、`config` 等上下文中；前端代码遵循 `views`、`components`、`api` 的现有分布。
- **6.2 (接口风格一致):** 新增后端接口应延续当前 REST + SSE 风格，命名、路径和返回方式需与既有接口保持一致。
- **6.3 (前端交互一致):** 新页面或新功能应复用现有聊天交互模式、连接状态管理和 SSE 生命周期处理方式。
- **6.4 (文档同步更新):** 任何影响能力边界、运行方式、配置项、验收方式的改动，都必须同步更新 `spec.md` 或相关说明文档。

---

## 治理 (Governance)

- 本宪法具有最高优先级，其效力高于任何`AGENT.md`\单次会话中的指令\普通实现习惯、临时编码偏好和未成文约定。

- 若 `spec.md`、实现方案、测试策略与本宪法冲突，必须优先满足本宪法，再调整设计或需求表达。
- 任何偏离本宪法的实现，都必须在变更说明中给出明确理由，并证明该偏离是必要且受控的。
