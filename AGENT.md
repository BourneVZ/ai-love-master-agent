# ai-love-master-agent 协作手册

@./constitution.md

你正在协助开发 `ai-love-master-agent` 项目。
在分析需求、设计方案、修改代码、补充测试和更新文档之前，必须先遵守 `constitution.md` 中定义的原则。

## 项目定位

这是一个基于 AI 能力构建的全栈应用项目，核心关注点不是传统 CRUD，而是：

- AI 对话应用
- Agent 能力编排
- Tool Calling
- RAG
- MCP 扩展
- 流式交互体验

## 技术上下文

- 后端：Java 21、Spring Boot、Spring AI
- 前端：Vue 3、Vite
- 构建：Maven、npm
- 关键能力：SSE、Tool Calling、RAG、MCP

## 工作方式

- 在动手前，先阅读相关代码和现有文档，不要脱离仓库现状凭空设计。
- 优先复用现有模式、现有结构和现有依赖，不轻易引入新抽象或新框架。
- 优先做小步、可验证的修改，避免大范围重写。
- 所有改动都应与 `spec.md` 保持一致；如果发现实现、文档与需求不一致，先明确差异，再决定修改方向。

## 实现要求

- 保持代码清晰、依赖显式、边界明确。
- 优先遵循现有项目分层，不把不属于当前层的逻辑塞进同一个类或组件。
- 对 AI、工具、检索、流式输出等能力，优先保持可插拔、可替换、可单独演进。
- 涉及外部服务、密钥、模型、向量库或 MCP 时，必须通过配置管理，不能写死在代码里。

## 测试与验证

- 新功能、行为变更和缺陷修复都必须有对应的验证方式。
- 优先补充贴近真实链路的测试，而不是只做形式化 Mock。
- 无法自动化验证时，至少补充明确的手工验收步骤。

## 风险控制

- 文件操作、命令执行、资源下载、外部输入处理都属于高风险能力，修改时必须优先考虑安全边界。
- 当依赖不可用、行为不确定或风险不清晰时，优先返回明确失败，而不是猜测性执行。

## 常用命令

- 后端测试：`mvn test`
- 后端构建：`mvn clean package`
- 后端运行：`mvn spring-boot:run`
- 前端安装依赖：`npm install`
- 前端开发：`npm run dev`
- 前端构建：`npm run build`

## 输出要求

- 说明问题时尽量基于实际代码和文件，而不是抽象猜测。
- 说明改动时，清楚写明改了什么、为什么改、如何验证。
- 如果存在阻塞，直接说明缺少的前置条件或外部依赖。
