# ai-love-master-agent 需求规格说明

## 1. 文档目的

本文档基于仓库 `README.md`、后端 Spring Boot 源码、前端 Vue 源码以及现有测试类整理，目标是将当前项目的业务目标、功能边界、工程约束和验收方式编译为可执行的需求规格。

本文档区分两类范围：

- 当前已实现并已对外暴露的能力：可通过前端页面或后端接口直接使用。
- 当前代码中已具备实现基础、但未全部在前端显式开放的能力：如 RAG、结构化输出、MCP 扩展能力，属于项目的明确功能目标或预留扩展点。

## 2. 项目概述

`ai-love-master-agent` 是一个以 Spring AI 为核心的 AI 智能体示例项目，包含两个主要业务场景：

- AI 恋爱大师：面向情感咨询场景的多轮对话应用，支持会话记忆，并预留 RAG、工具调用、MCP 调用等增强能力。
- AI 超级智能体 LoveManus：面向复杂任务执行的自主规划 Agent，能够在多步推理中组合使用搜索、网页抓取、文件、下载、终端、PDF 等工具。

系统由以下部分组成：

- 后端：Spring Boot 3 + Spring AI，负责模型调用、工具注册、SSE 流式输出、Agent 编排。
- 前端：Vue 3 + Vite，提供首页、恋爱大师对话页、超级智能体对话页。
- MCP 子项目：`image-search-mcp-server`，用于提供图片搜索类 MCP 服务能力。
- 文档知识库：`src/main/resources/document` 下的恋爱场景知识文档，为 RAG 能力提供语料基础。

## 3. 角色与用户

### 3.1 目标用户

- 普通终端用户：希望通过 AI 获得恋爱/婚姻情感建议，或希望让 AI 帮忙完成复杂任务。

### 3.2 外部依赖角色

- 大模型服务提供方：当前默认接入 DashScope。
- 搜索服务提供方：SearchAPI。
- 可选的 MCP 服务提供方：图片搜索服务、地图服务等。

## 4. 用户故事

### 4.1 AI 恋爱大师

- 作为单身用户，我希望向 AI 描述我的情感困惑，以便获得针对性的建议，而不是通用鸡汤。
- 作为恋爱中的用户，我希望 AI 记住我的上下文信息，以便连续几轮追问时不需要重复背景。
- 作为已婚用户，我希望 AI 能结合婚恋知识库回答问题，以便获得更贴近场景的建议。
- 作为用户，我希望在回答较长时看到流式输出，以便减少等待焦虑。
- 作为用户，我希望在 AI 有需要时能调用搜索、网页抓取或其他工具辅助回答，以便提高建议的可操作性。

### 4.2 AI 超级智能体

- 作为用户，我希望提交一个复杂目标，例如“制定约会计划并导出 PDF”，让智能体自主拆解步骤执行。
- 作为用户，我希望看到智能体逐步执行的过程输出，以便理解它做了什么。
- 作为用户，我希望智能体在必要时能调用多种工具串联完成任务，而不是只返回纯文本建议。
- 作为用户，我希望智能体在任务完成或达到最大步数后能安全结束，而不是无限循环。

### 4.3 开发与运维

- 作为开发者，我希望工具能力通过统一注册机制接入，以便后续扩展工具不影响核心业务逻辑。
- 作为开发者，我希望前后端通过稳定接口和 SSE 约定解耦，以便替换 UI 或扩展业务页。
- 作为运维人员，我希望系统提供健康检查接口，以便部署后能快速判断服务是否存活。

## 5. 功能性需求

### 5.1 首页与导航

- 系统应提供首页，展示两个主入口：
  - `AI 恋爱大师`
  - `AI 超级智能体`
- 用户点击入口后应跳转到对应对话页面。

### 5.2 AI 恋爱大师对话

#### 5.2.1 基础对话

- 系统应提供恋爱大师同步对话接口：`GET /api/ai/love_app/chat/sync`。
- 系统应提供恋爱大师流式对话接口：`GET /api/ai/love_app/chat/sse`。
- 系统应支持基于 `chatId` 的多轮会话记忆。
- 系统应使用预设系统提示词，将 AI 角色限定为恋爱情感咨询顾问。

#### 5.2.2 SSE 输出

- 系统应支持基于 `Flux<String>` 的 SSE 文本流式输出。
- 系统应额外提供 `ServerSentEvent` 和 `SseEmitter` 两种流式接口形态，以适配不同前端实现方式：
  - `GET /api/ai/love_app/chat/server_sent_event`
  - `GET /api/ai/love_app/chat/sse_emitter`
- 前端应在接收到流式片段时实时拼接 AI 回复内容。

#### 5.2.3 会话记忆

- 同一 `chatId` 下的连续对话应保留最近窗口内的消息上下文。
- 当前实现中消息窗口上限应为 20 条。
- 不同 `chatId` 之间的会话内容应相互隔离。

#### 5.2.4 结构化输出

- 系统应支持生成恋爱报告，报告至少包含：
  - 标题
  - 建议列表
- 当前实现通过 `LoveReport` 结构体承载该能力，属于后端能力，前端尚未直接暴露。

#### 5.2.5 RAG 问答

- 系统应支持基于恋爱知识库的问答增强。
- 系统应在检索前支持查询改写，以提升召回效果。
- 系统应能够从本地向量库、云端向量服务或 PgVector 方案中选择一种或多种方式进行增强。
- 当前源码中已包含本地 `VectorStore`、云端 Advisor、PgVector、自定义 RAG Advisor 等实现入口，但运行时是否启用依赖具体配置。

#### 5.2.6 工具调用增强

- 系统应允许恋爱大师在回答问题时调用已注册工具。
- 工具调用后的结果应参与最终回复生成。
- 当前后端支持的工具包括：
  - 文件读写
  - Web 搜索
  - 网页抓取
  - 资源下载
  - 终端命令执行
  - PDF 生成
  - 终止工具

#### 5.2.7 MCP 调用增强

- 系统应支持通过 `ToolCallbackProvider` 接入 MCP 工具。
- 当前项目中 MCP 能力的典型目标包括图片搜索、地图类服务。
- 当前能力是否可用依赖：
  - `spring.ai.mcp.client` 配置
  - MCP 子服务是否已启动

### 5.3 AI 超级智能体 LoveManus

#### 5.3.1 任务输入与执行

- 系统应提供超级智能体流式接口：`GET /api/ai/manus/chat`。
- 用户应能够输入开放式复杂任务目标。
- 智能体应以多步执行方式完成任务，而不是单次直接回答。

#### 5.3.2 自主规划机制

- 智能体应具有固定系统提示词与下一步行动提示词。
- 智能体应在每一步中自主判断是否调用工具。
- 若模型返回工具调用，系统应执行工具，并将工具返回结果写回上下文。
- 若模型未返回工具调用，系统应将助手消息写入上下文，并结束执行。

#### 5.3.3 步数控制

- 智能体应支持最大执行步数限制，防止无穷循环。
- 当前实现中 `LoveManus` 最大步数应为 20。
- 当达到最大步数时，系统应终止执行并返回明确结束信息。

#### 5.3.4 终止机制

- 系统应提供终止工具供模型主动结束流程。
- 当检测到终止工具被调用时，智能体状态应切换为 `FINISHED`。

#### 5.3.5 流式过程反馈

- 智能体应以步骤为粒度通过 `SseEmitter` 持续输出执行结果。
- 前端应将 SSE 片段按句子或长度阈值聚合为消息气泡，以提升可读性。

### 5.4 工具系统

#### 5.4.1 工具注册

- 系统应通过集中注册类统一创建和注入工具集合。
- 工具集合应以 `ToolCallback[]` 暴露给业务应用和智能体使用。

#### 5.4.2 文件操作工具

- 系统应支持读取指定文件内容。
- 系统应支持将文本写入指定文件。
- 文件应保存到统一的业务目录下。

#### 5.4.3 Web 搜索工具

- 系统应支持根据用户关键词调用外部搜索 API。
- 当前实现应基于 SearchAPI 调用百度搜索引擎并返回前 5 条自然搜索结果。

#### 5.4.4 网页抓取工具

- 系统应支持抓取指定 URL 的 HTML 内容。

#### 5.4.5 资源下载工具

- 系统应支持根据 URL 下载文件到本地目录。

#### 5.4.6 终端操作工具

- 系统应支持执行终端命令并返回标准输出文本。
- 当前实现基于 `cmd.exe /c` 执行命令，默认面向 Windows 环境。

#### 5.4.7 PDF 生成工具

- 系统应支持将文本内容生成为 PDF 文件。
- 生成 PDF 时应支持中文字体输出。

### 5.5 健康检查与基础运维

- 系统应提供健康检查接口：`GET /api/health`。
- 健康检查成功时应返回 `ok`。

### 5.6 前端交互

- 前端应展示对话消息列表、输入框、发送按钮和连接状态。
- 当连接处于 `connecting` 状态时，输入框和发送按钮应禁用，防止重复发送。
- 恋爱大师页应显示会话 ID。
- 页面卸载时应主动关闭 SSE 连接，避免资源泄漏。

## 6. 非功能性需求

### 6.1 架构解耦

- 前端与后端必须通过 HTTP/SSE 接口交互，不直接耦合模型实现细节。
- 控制器层只负责接口暴露与协议转换，不直接承载复杂业务编排。
- 业务能力应封装在应用/Agent 层：
  - `LoveApp` 负责情感咨询应用编排
  - `LoveManus` 负责自主规划 Agent 编排
- 工具能力应独立封装为单一职责类，通过 `ToolRegistration` 统一注册，避免业务代码直接 new 多个具体工具。
- RAG、MCP、结构化输出等增强能力应作为可插拔模块存在，不应强耦合到基础聊天链路。
- 向量库、模型、外部搜索服务应通过配置注入切换，避免将供应商能力硬编码在控制器和前端中。

### 6.2 错误处理

- 当模型调用、工具调用、网络请求、文件读写或命令执行失败时，系统应返回明确错误信息，而不是静默失败。
- SSE 流式接口在异常时应结束连接或通过错误回调通知前端。
- 智能体执行过程中若出现异常，应进入错误状态并输出错误说明。
- 工具层应自行捕获异常并返回错误描述文本，防止单个工具异常导致服务进程崩溃。
- 当前前端在 SSE 失败后应切换到 `error` 状态并关闭连接。

### 6.3 可配置性

- 模型 API Key、Search API Key、MCP 配置、向量库配置必须通过配置文件或环境变量注入。
- 开发环境与生产环境应支持分 profile 配置。

### 6.4 性能与响应

- 普通聊天场景应优先使用流式输出，提升首字返回时间体验。
- 智能体长任务执行时应逐步输出中间结果，避免长时间无响应。
- 会话记忆窗口应有限制，避免上下文无限增长导致成本和响应时间失控。

### 6.5 可维护性

- 系统应保留测试类覆盖核心场景：聊天、RAG、工具调用、MCP 调用、Agent 执行。
- 日志应覆盖模型回答、思考过程、工具调用过程，便于排查问题。

### 6.6 安全性

- 终端工具、文件工具、下载工具属于高风险能力，生产环境必须增加更严格的访问限制、命令白名单、目录白名单与审计机制。
- 外部网页抓取和下载能力必须防止恶意 URL、超大文件、超时和不可访问资源影响服务稳定性。
- API Key 不得硬编码为真实值提交到仓库。

## 7. 约束与假设

- 默认运行环境为 Java 21、Node.js 前端构建环境以及 Windows 命令执行环境。
- 项目强依赖外部大模型和第三方服务，未配置有效 API Key 时，聊天和工具增强能力无法完整验证。
- RAG、PgVector、MCP 等增强能力依赖额外配置或外部服务启动，不保证开箱即用。
- 当前前端只直接调用：
  - `GET /api/ai/love_app/chat/sse`
  - `GET /api/ai/manus/chat`
- 当前后端虽提供同步、结构化输出、RAG、MCP 等能力，但前端未全部封装为显式页面功能。

## 9. 当前风险与改进建议

- 当前前端主要消费基础聊天和 LoveManus 流式接口，RAG、结构化输出、MCP 等能力尚未形成完整可视化入口。
- `TerminalOperationTool` 当前可执行任意命令，适合作为教学/实验功能，不适合直接暴露到生产环境。
- 工具层多数以字符串返回错误信息，便于演示，但不利于前端做结构化错误处理，后续可统一错误模型。
- `README` 描述的能力范围大于当前前端已开放范围，后续应在产品层明确“已交付功能”和“实验能力”。

## 10. 建议的后续迭代

- 为恋爱大师补充 RAG、报告生成、MCP 调用的独立接口与前端开关。
- 为工具调用增加权限控制、超时控制、白名单控制和审计日志。
- 为 Agent 输出定义结构化事件协议，而不是纯文本步骤流。
- 为核心链路补充端到端自动化测试，覆盖 SSE、对话记忆、工具执行与异常路径。
